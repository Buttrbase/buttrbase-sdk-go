## Design Decision: TokenClaims approach
`jwt.ParseWithClaims` requires the claims struct to implement the `jwt.Claims` interface (6 getter methods). The public `TokenClaims` has `Exp int64` / `Iat int64` which conflict with `jwt.RegisteredClaims`'s `ExpiresAt *NumericDate` / `IssuedAt *NumericDate`. Rather than alter the public struct (which would break existing JSON deserialization and all existing tests), an unexported `jwtClaims` type embeds `jwt.RegisteredClaims` plus buttrbase-specific fields. After `jwt.ParseWithClaims` succeeds, `jwtClaims.toTokenClaims()` converts to the existing public `TokenClaims`. Callers see no new public types; `TokenClaims.AuthContext()` works unchanged.

## Concerns
- The background JWKS refresh goroutine logs `slog.ErrorContext` when the URL is unreachable (visible in test output for the 4 tests that use `localhost:9999` as a dummy URL). This is expected behaviour from keyfunc — the tests pass immediately because they exercise paths that never reach the JWKS fetch. In production the URL will always be reachable.
- `NewVerifier` (no context) runs the refresh goroutine for the process lifetime; callers that need clean shutdown should use `NewVerifierCtx`.
