# Task D Report — token-claims-enrichment

## Status
COMPLETE

## What was done

Added claims-enrichment types and helpers to the Go SDK, additively mirroring
the Rust SDK 0.6.0 changes in `src/verify/verifier.rs`.

### New file: `claims.go`
- `TokenClaimsData` — the `data` envelope from buttrbase JWTs (`roles *string`,
  `email *string`, `org_uuid *string`, `user_uuid *string`; all optional with
  `omitempty`).
- `TokenClaims` — full JWT payload (`sub`, `org`, `exp`, `iat`, `scope []string`,
  `data *TokenClaimsData`).
- `AuthContext` — derived principal (`UserID`, `OrgID`, `Scopes []string`,
  `Roles []string`, `Email *string`).
- `(TokenClaims).AuthContext()` — splits `data.roles` (comma/space-delimited)
  into `Roles []string`; forwards `data.email`. Empty `data` → empty slice /
  nil email.
- `ParseTokenClaims(tokenString)` — base64url-decodes the JWT payload and
  unmarshals it. **Does not verify the signature** (SDK has no JWKS cache).
  Documented clearly.

### New file: `claims_test.go`
11 tests covering:
- `TokenClaimsData` deserialization from the canonical fixture JSON
  (`access_token_claims.json` shape: `data.roles = "owner"`,
  `data.email = "test@example.com"`).
- `TokenClaims` top-level field mapping.
- `AuthContext` derivation: single role, multiple comma-delimited roles,
  space-delimited roles, no data envelope, empty roles string.
- `ParseTokenClaims`: happy path with a realistic fake JWT (fixture payload
  base64url-encoded), malformed tokens (empty, 2-part, 4-part, bad base64),
  invalid JSON payload.

## Concerns
- `ParseTokenClaims` does not verify the signature. This is intentional and
  documented (the Go SDK is a REST API client, not a JWKS verifier; the Rust
  SDK has a full `Verifier` with JWKS cache). Callers should only use this
  after verifying the token externally.
- The fixture JWT embedded in tests uses a fake header/signature to avoid
  pulling in a JWT-signing dependency. The payload decodes correctly.
